## 1. Origin: it was a pivot, not a master plan

VS Code started as **Monaco**, a browser code editor (Erich Gamma joined Microsoft in 2011 to build it). It became "Visual Studio Online" for editing Azure sites — and *failed*, ~3,000 monthly users. They pivoted it into a **desktop** product, shipped at Build 2015, and went open source 6 months later at Connect 2015. The lesson: it found its identity by narrowing scope (browser → desktop editor) and opening up at the same time.

## 2. The architecture: thin core, fat extensions

This is the part most worth copying.

- **Multi-process**: a main process, renderer processes for UI, and a separate **extension host** — one per window. Extensions run isolated in a Node.js process so a misbehaving extension can't freeze the UI. Communication is a bidirectional RPC bridge.
- **Language Server Protocol (LSP)**: the editor doesn't know any language. A language server speaks a standard protocol, in *any* language. This decoupling is arguably *the* reason VS Code won — community can add language support without touching the core.
- **"Thin core" philosophy**: the core stays small and tightly controlled; nearly everything interesting is an extension.

## 3. The dual identity — this is the real model for "personal + community"

VS Code is genuinely two things:

- **Code – OSS** — MIT licensed, on GitHub. Anyone can read it, fork it, build it. This is the *community* artifact.
- **Visual Studio Code** — the build Microsoft ships, under a *proprietary* Microsoft license. It adds branding (logo, name), telemetry, and Marketplace access. Only ~a small % of code differs.

The lever that keeps the official build dominant: the **Extension Marketplace ToS** forbids forks (like VSCodium) from using it. So: *community owns the open core; the owner owns the brand, the official build, and the distribution channel.*

## 4. Community governance is deliberately bounded

- **CLA** required before any PR is accepted (one-time).
- Issues triaged with `help wanted` / `good first issue` labels — community work is *curated*, not open-ended.
- **"No external contributions accepted"** for certain components — some areas stay owner-only.

# 1. The module surface

This is the big one. It answers: when someone "adds to" terminite — a module, or the AI itself — **where does that code run, and how does it talk to terminite's core?**

Two answers:

### Out-of-process protocol
The module runs as its **own separate program**, and talks to terminite over a socket — two independent programs having a structured conversation. This is how tmux control mode works, and how VS Code's extensions work.

*Analogy: terminite is a restaurant; a module is a separate food truck that coordinates with it by radio. Independent businesses, cooperating.*

- ✅ **Any language.** A module just has to speak the protocol — Rust, TypeScript, Python, anything. This is where community love lives.
- ✅ **Crash isolation.** A broken module crashes itself, not your terminal.
- ✅ **The AI fits for free.** The AI becomes "just another client" of the same protocol as modules. Your whole thesis — the AI as a co-user — is served by the *same* mechanism as modularity. One thing, not two.
- ❌ **Heavier to build.** You have to design the protocol — every message, versioned, serialized.
- ❌ A hair slower per call (crossing a process boundary). Invisible for terminal work — tmux and VS Code prove it.

### In-process Rust modules
The module is **compiled into terminite itself** and calls its functions directly.

*Analogy: instead of a food truck, you hire a new chef into the kitchen. Same building, same memory.*

- ✅ **Lighter, faster, simpler.** No protocol — just Rust function calls.
- ❌ **Rust-only.** The entire non-Rust community is locked out of the module layer — the exact opposite of the VS Code lesson.
- ❌ **No crash isolation.** A bad module takes the whole terminal down.
- ❌ **No seam for the AI.** The AI isn't a Rust library — so you'd have to build the AI's mechanism *separately*. Two things instead of one.

**My recommendation: out-of-process.** It's the only option where all three things terminite exists for — *add to it*, *serve the AI*, *be loved by the community* — are one mechanism instead of three. The cost is real (you build a protocol), but you build *one* thing, and it's the thing that makes terminite *terminite*. Pragmatic note: you can design it protocol-shaped and implement the transport incrementally — start simple, the socket comes later.

# 2. The GPU layer

This is small — contained entirely inside the Renderer. It answers: **how does terminite's drawing code talk to the graphics card?**

The catch: every OS has its own GPU API — macOS uses **Metal**, Linux uses Vulkan, Windows uses Direct3D.

### wgpu
A Rust library that is *one* GPU API working everywhere — you write rendering code once, wgpu translates it to Metal / Vulkan / Direct3D underneath.

*Analogy: a universal travel adapter. One plug, fits every country's socket.*

- ✅ Write once, runs on every platform's GPU — keeps the cross-platform door open.
- ✅ Modern, well-maintained, the Rust-ecosystem default.
- ❌ A thin layer between you and Metal; tiny overhead.

### Metal directly
Skip the adapter — write straight against Apple's own GPU API.

*Analogy: wire the plug for the local socket directly. Perfect fit — one country only.*

- ✅ Maximum control and performance on macOS.
- ❌ **macOS-only.** Wanting Linux/Windows later means rewriting the entire renderer — re-slamming the door you just kept open.

**My recommendation: wgpu.** Two reasons: (1) it keeps the cross-platform door open, at the rendering layer — the *hardest* layer to retrofit. (2) A terminal is a *light* GPU workload — a grid of text cells, nothing like a 3D game. wgpu's overhead vanishes; you hit the loveliness bar easily. Direct Metal would buy performance you don't need at the cost of portability you might.
